4. Requerimientos Funcionales

A continuación se detallan las funcionalidades principales del sistema SIGP, diseñado bajo arquitectura MVC.

Funcionalidad Gestión de Usuarios y Roles (Auth)
RF-001 RF-001 Versión 1.0
Precondición El usuario no debe estar logueado. Base de datos operativa.
Descripción El sistema permitirá el acceso diferenciado para tres roles: Administrador (CCPE), Empresa y Estudiante.
Secuencia de Acción 1. Usuario ingresa a la URL del login.
2. Ingresa credenciales (correo y contraseña).
3. Sistema valida credenciales encriptadas.
4. Sistema redirige al Dashboard correspondiente según el rol.
Postcondición Usuario autenticado con acceso a su menú específico.
Excepciones Credenciales inválidas: mostrar alerta "Usuario o contraseña incorrectos".
Funcionalidad Publicación de Ofertas de Pasantía
RF-00X RF-002 Versión 1.0
Precondición Usuario Empresa autenticado y validado por CCPE.
Descripción Las empresas podrán crear fichas de ofertas detallando perfil, actividades y horarios.
Secuencia de Acción 1. Empresa selecciona "Nueva Oferta".
2. Completa formulario (Título, Descripción, Requisitos, Vacantes).
3. Guarda la oferta.
4. Sistema notifica al Admin para revisión.
Postcondición Oferta creada en estado "Pendiente de Aprobación".
Excepciones Campos vacíos: El sistema resalta los campos obligatorios.
Funcionalidad Postulación a Pasantías
RF-00X RF-003 Versión 1.0
Precondición Usuario Estudiante autenticado y perfil completo.
Descripción Los estudiantes pueden aplicar a las ofertas activas enviando su perfil digital.
Secuencia de Acción 1. Estudiante visualiza oferta.
2. Clic en "Aplicar".
3. Sistema confirma envío de datos y CV adjunto.
4. Se registra la postulación en la base de datos.
Postcondición Empresa recibe notificación de nuevo candidato. Estado "Postulado".

5. Requerimientos No Funcionales

1. Requisitos de Desempeño

    [cite_start]
  • El sistema debe cargar el Dashboard principal en menos de 2 segundos. [cite: 24]
  • El sistema debe soportar la concurrencia de al menos 50 usuarios simultáneos (estudiantes/empresas).

2. Requisitos de Usabilidad

  • El diseño debe ser Responsive (adaptable a móviles y tablets) utilizando Bootstrap o CSS Grid.
  • El proceso de registro de estudiante no debe tomar más de 5 minutos.

3. Requisitos de Confiabilidad

    [cite_start]
  • Disponibilidad del servicio del 99.9% durante el periodo de aplicaciones (Enero-Marzo). [cite: 35]
  • Backups automáticos de la base de datos semanalmente.

4. Requisitos de Seguridad

    [cite_start]
  • Las contraseñas deben almacenarse encriptadas (ej. BCRYPT o Argon2). [cite: 39]
  • Protección contra inyección SQL mediante el uso de PDO/Prepared Statements en PHP.
  • Validación de sesiones para evitar acceso a URLs directas sin login.
